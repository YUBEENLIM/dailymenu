# Business Logic — 도메인 규칙 & 추천 정책

## 1. 서비스 개요

**메뉴 추천 서비스**는 직장인을 대상으로 점심/저녁 시간대 위치 기반 + 식사 이력 기반 맞춤 메뉴를 추천한다.

### 트래픽 특성

| 구분 | 시간대 | 비중 |
|---|---|---|
| 점심 피크 | 11:30 ~ 12:30 | 전체 트래픽의 ~40% |
| 저녁 피크 | 17:30 ~ 18:30 | 전체 트래픽의 ~30% |
| 평시 | 그 외 | ~30% |

**피크 기준으로 시스템을 설계한다. 평균 TPS로 설계하면 피크에 반드시 장애가 발생한다.**

---

## 2. 핵심 Context & 역할

### 추천 이력 vs 식사 이력 구분 (자주 혼동)

| 구분 | 주체 | 의미 | 용도 |
|---|---|---|---|
| 추천 이력 | 시스템 | 시스템이 권한 메뉴 기록 | 같은 메뉴 연속 추천 방지 |
| 식사 이력 | 사용자 | 사용자가 실제 먹은 기록 | 다양성 필터 적용 |

> "시스템이 권한 것" ≠ "사용자가 먹은 것"

### 이력 우선순위

식사 이력 > 추천 이력

실제로 먹은 데이터가 항상 우선한다.
추천 점수 계산 시 식사 이력을 먼저 적용하고, 추천 이력은 보조 기준으로 사용한다.

### 데이터 신선도 기준

| 데이터 | 허용 기준 | 이유 |
|---|---|---|
| 외부 API 응답 캐시 | 30분 이내 | API 호출 비용 절감 |
| 추천 결과 캐시 | 2분 이내 | 동일 요청 재계산 방지 |
| 식사 이력 / 추천 이력 | 항상 DB 기준 | 강한 일관성 필요 |

신선도가 떨어진 데이터는 Fallback Level 상승 판단에 영향을 준다.

### 추천 결과 캐시 사용 조건

다음 조건을 모두 만족할 때만 캐시된 추천 결과를 재사용한다.

- 동일 사용자의 요청
- 동일 위치 (오차 범위 이내)
- 2분 이내 생성된 추천 결과

### 외부 장소 수집/동기화 Context

현재: 카카오맵 API (`KakaoPlaceAdapter`)
예정: 네이버맵 API (`NaverPlaceAdapter`), 공공데이터 API

**어댑터 추가 시 `PlacePort` 인터페이스를 구현하기만 하면 된다. 도메인 로직은 변경하지 않는다.**

데이터 동기화 전략:
- 서비스 출시 전: 수동 시드 데이터 구축 (식당, 메뉴, 카테고리, 위치, 운영 기본 정보)
- 운영 중: 주기적 배치로 자동 동기화 (폐업, 메뉴 변경 반영)

## 3. 추천 흐름 (Happy Path 10단계)

```
1.  사용자 → [추천 받기] 클릭
2.  추천 Context → 사용자 프로필 Context 조회
        (이 사람이 어떤 취향인지 파악)
3.  추천 Context → 식사 이력 Context 조회
        (최근에 실제로 무엇을 먹었는지 확인)
4.  추천 Context → 추천 이력 Context 조회
        (최근에 무엇을 추천했는지 확인)
5.  추천 Context → 장소/위치 Context 조회
        (사용자 현재 위치 기준 도달 가능 식당 후보군 필터링)
         → PlacePort → KakaoPlaceAdapter (외부 API 호출)
6.  추천 Context → 식당/메뉴 카탈로그 Context 조회
        (위치 기준으로 걸러진 식당들의 실제 메뉴 데이터 조회)
7.  추천 정책 적용 (Domain 순수 로직)
8.  추천 결과 생성
9.  추천 이력 저장 (무엇을 추천했는지 기록)
10. 사용자에게 추천 결과 반환
```

**병렬 조회 가능 구간:** 2번(사용자 프로필), 3번(식사 이력), 4번(추천 이력)은 서로 의존하지 않으므로 병렬 조회 검토 가능.

## 4. 추천 정책 (RecommendationPolicy)

### 다양성 필터 — 실험 그룹

| 그룹 | 정책 |
|---|---|
| A | 최근 2일 같은 메뉴 제외 |
| B | 최근 3일 같은 메뉴 제외 |
| C | 최근 2일 같은 메뉴 + 같은 카테고리도 제한 |

### 탐색-활용 균형 (Exploration-Exploitation)

- 90%: 검증된 선호 기반 추천
- 10%: 새로운 메뉴 탐색 추천

### 사용자 프로필 기반 필터

- 혼밥 가능 여부
- 건강 기준 (저칼로리, 채식 등)
- 가격 범위
- 선호 카테고리

### 필터 적용 우선순위

```
1. 위치 기반 필터 (도달 불가능한 식당 제거)
2. 식사 이력 기반 다양성 필터
3. 사용자 프로필 필터 (건강, 가격, 혼밥)
4. 추천 이력 기반 중복 방지
5. 탐색-활용 비율 적용
```

### 추천 점수 계산 (초안)

추천 점수는 0~100점 범위로 계산한다.
초기에는 단순한 요소만 반영하고, 사용자 데이터가 쌓이면 가중치를 조정한다.

**거리 (30점)**

| 조건 | 점수 |
|---|---|
| 500m 이내 | 30점 |
| 500~700m | 20점 |
| 700~1000m | 10점 |
| 1000m 초과 | 0점 |

**선호 카테고리 (30점)**

| 조건 | 점수 |
|---|---|
| 선호 카테고리 일치 | 30점 |
| 무관심 카테고리 | 15점 |
| 비선호 카테고리 (user_restrictions 기준) | 0점 |

**최근 식사 이력 (30점)**

| 조건 | 점수 |
|---|---|
| 3일 이내 먹지 않음 | 30점 (만점) |
| 3일 전 먹음 | 20점 |
| 2일 전 먹음 | 10점 |
| 1일 전 먹음 | 0점 |
| 당일 먹음 | 추천 제외 |

**추천 이력 (10점)**

| 조건 | 점수 |
|---|---|
| 3일 이내 추천 안 됨 | 10점 (만점) |
| 3일 전 추천됨 | 7점 |
| 2일 전 추천됨 | 3점 |
| 1일 전 추천됨 | 0점 |
| 당일 추천됨 | 추천 제외 |

> 혼밥 가능 여부, 건강 기준은 점수에 반영하지 않고 필터로만 사용한다.
> 사용자 반응 데이터(수락률, 거절 사유)가 쌓이면 가중치를 조정한다.

### 점수 동일 시 우선순위

추천 점수가 동일할 경우 다음 기준으로 정렬한다.

1. 거리 가까운 순
2. 최근 추천되지 않은 순
3. 랜덤 (탐색 다양성 확보)

### 추천 실패 정의

다음 조건 중 하나라도 만족하면 추천 실패로 간주한다.

- 후보 메뉴 0개 (Fallback Level 2까지 수행 후에도 없음)
- 외부 API 및 캐시 데이터 모두 조회 실패
- 추천 점수 계산 불가 (필수 데이터 부족)

### 결과 보장 원칙

추천 요청은 가능한 한 최소 1개의 결과를 반환한다.

- Level 2까지는 반드시 1개 이상 반환을 목표로 한다
- Level 3에서도 최소 추천 리스트를 유지한다
- Level 4에서만 추천 결과 0개 허용

### 추천 요청 처리 제한

추천 요청은 최대 5초 이내 응답해야 한다.
5초 초과 시 즉시 Fallback 단계로 전환한다.

## 5. Fallback 정책 (4단계)

추천 품질이 저하되거나 외부 API에 장애가 생기면 아래 순서로 Fallback한다.
**핵심 원칙: 장애가 나더라도 추천 경험 자체가 완전히 끊기지 않는 것이 목표다.**

### Level 1 — Best Effort (캐시 기반)

**상황:** 실시간 데이터 불러오기 지연
**처리:** 최근 성공 결과나 캐시 기반 추천 반환 (핵심 필터 유지)
**UX 메시지:** "실시간 데이터 확인이 잠시 지연되어 최근 기준으로 메뉴를 추천해드렸어요."

### Level 2 — Reduced Function (필터 완화)

**상황:** 추천 후보 부족 또는 맞춤 분석 불가
**처리:** 조건 순차 완화 (최근 제외 기간 단축 → 거리 범위 확장 → 건강 필터 완화), 최소 1개 이상 반드시 반환
**UX 메시지:** "지금은 맞춤 분석이 원활하지 않아, 일부 조건을 완화해서 추천해드렸어요."

### Level 3 — Non-personalized (비개인화 추천)

**상황:** 추천 정확도 보장 불가, 개인화 데이터 활용 불가
**처리:** 추천 엔진 대신 사전에 적재된 인기 메뉴 / 대표 메뉴 / 시간대별 추천 리스트 기반 비개인화 추천 제공
**UX 메시지:** "현재 맞춤 추천이 잠시 어려워서 점심에 인기 있는 메뉴를 보여드릴게요."

> Level 3 데이터는 사전 적재(배치) 기반이므로 실시간 장애에 영향받지 않아야 한다. 적재 데이터마저 없으면 Level 4로 내려간다.

### Level 4 — Minimal UI (추천 생성 중단)

**상황:** 데이터 소스가 완전히 죽어 추천 생성 자체가 불가능한 경우
**처리:** 추천 생성 중단, 메뉴 카테고리 탐색 / 재시도 / 즐겨찾기 등 최소 선택 기능 제공
**UX 메시지:** "지금은 추천이 어려운 상황이에요. 카테고리에서 직접 찾아보시거나, 잠시 후 다시 시도해주세요."

> Level 4에서도 빈 화면은 절대 보여주지 않는다. 사용자가 다음 행동을 할 수 있는 최소한의 UI는 항상 제공해야 한다.

## 6. 멱등성 & 중복 요청 처리

| 상황 | 응답 |
|---|---|
| 처음 요청 | 정상 처리 |
| 처리 중 중복 요청 | 409 Conflict |
| 완료 후 중복 요청 | 최초 처리 결과 동일 반환 |
| 다른 내용의 같은 키 | 요청 해시 비교 후 충돌 처리 |

## 7. 사용자 반응 데이터 수집

추천 품질 개선을 위해 아래 반응을 수집한다.

| 반응 | 의미 |
|---|---|
| 추천 메뉴 클릭 | 관심 있음 |
| 추천 확정 | 수락 |
| 추천 거절 | 거절 (사유 수집) |
| 다시 추천 요청 | 불만족 (사유 수집) |
| 아무 행동 없이 이탈 | 무관심 |

재추천 시 사유 수집 예시: "너무 멀어요", "오늘은 가볍게 먹고 싶어요", "혼자 가기 애매해요"

## 8. 가설 검증 원칙

이 서비스의 핵심 가설은 **"추천을 대신 해주면 사용자의 선택 피로가 줄어든다"** 는 것이다.
가설이 틀렸다면 아무리 정교한 추천 로직도 의미가 없다.

**기능을 정교하게 만들기 전에, 이 기능이 사용자 문제를 실제로 해결하는지 먼저 검증해라.**
검증되지 않은 가설 위에 복잡한 구현을 올리지 마라.

### 핵심 지표 — 추천 수락률

이 서비스의 "결제 완료율"에 해당하는 지표는 **추천 수락률**이다.

- 추천 수락률이 떨어지고 있다 → 추천 품질 문제 (알고리즘 개선)
- 재추천 요청이 반복된다 → 필터 조건 문제 (다양성 정책 재검토)
- 이탈률이 높다 → 신뢰 붕괴 가능성 (아래 신뢰 설계 참고)

### 개인화 전환 트리거

초반에는 다양성 우선, 이후 사용자 데이터 기반으로 개인화한다.
전환 기준은 **"감"이 아니라 데이터**로 결정한다.

| 조건 | 전환 내용 |
|---|---|
| 식사 확정 이력 10건 이상 | 선호 카테고리 가중치 반영 시작 |
| 재추천 사유 5건 이상 | 거절 패턴 분석 후 필터 조정 |
| 추천 수락률 60% 미만 지속 3일 | 다양성 정책 실험 그룹 재검토 |

---

## 9. 신뢰 설계 — 추천 3번 실패하면 떠난다

"대신 결정해준다"는 컨셉은 **사용자 신뢰** 위에서만 작동한다.
추천이 연속으로 마음에 안 들면 사용자는 "그냥 내가 고르는 게 낫겠다"로 돌아간다.

### 신뢰 붕괴 시나리오

- 방금 먹은 메뉴가 다시 추천됨 → 이력을 안 보는 서비스로 인식
- 10% 탐색 추천이 3번 연속 거절됨 → 이상한 메뉴를 추천하는 서비스로 인식
- 조건 완화 없이 빈 결과 반환 → 서비스가 안 된다고 인식

### 신뢰 방어 규칙

```
1. 방금 먹은 메뉴는 절대 다시 추천하지 않는다 (식사 이력 강한 일관성 유지)
2. 10% 탐색 추천이 연속 거절되면 해당 사용자의 탐색 비율을 일시적으로 낮춘다
3. 추천 결과가 없을 때 빈 화면을 절대 보여주지 않는다 (Fallback 3단계 필수)
4. 조건 완화가 발생했을 때 반드시 사용자에게 이유를 안내한다
```

### 데이터 품질 저하 감지

사용자가 초반에는 "먹었어요"를 누르다가 나중에 안 누르면 미확정 데이터만 쌓인다.
이 경우 추천 품질이 서서히 떨어지는데, 감지 기준은 아래와 같다.

| 신호 | 기준 | 대응 |
|---|---|---|
| 확정 이력 비율 급감 | 최근 7일 확정률 < 20% | 가벼운 넛지 UI 노출 |
| 재추천 요청 증가 | 재추천율 > 40% 지속 3일 | 추천 정책 점검 |
| 추천 후 이탈 증가 | 추천 결과 화면 이탈률 > 60% | 추천 결과 UX 점검 |
